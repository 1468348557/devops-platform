# DevOps Platform 操作手册

## 1. 手册说明

本文档用于指导管理员、运维/审批人员和研发人员使用 DevOps Platform。内容包括环境启动、系统初始化、账号权限、投产征集、HOBO 需求登记、新建分支、投产追板和 SQL 执行申请等操作。

## 2. 环境准备

### 2.1 基础依赖

- Python：建议使用与项目兼容的 Python 3 版本。
- 数据库：MySQL。
- Git：用于拉取业务仓库、检查分支、创建 MR 和 Tag。
- 依赖包：见 `myproject/requirements.txt`。

### 2.2 安装依赖

在仓库根目录执行：

```bash
cd myproject
pip install -r requirements.txt
```

### 2.3 配置环境变量

项目支持读取 `myproject/.env` 文件。常用配置示例：

```bash
DJANGO_ENV=dev
DJANGO_SECRET_KEY=replace-with-your-secret-key
DJANGO_DEBUG=true
DJANGO_ALLOWED_HOSTS=127.0.0.1,localhost

MYSQL_DATABASE=devops_platform
MYSQL_USER=devops
MYSQL_PASSWORD=replace-with-password
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
```

生产环境注意：

- `DJANGO_ENV` 建议设置为 `prod` 或 `production`。
- 必须设置非默认 `DJANGO_SECRET_KEY`。
- 必须设置 `MYSQL_PASSWORD`。
- 应正确配置 `DJANGO_ALLOWED_HOSTS`。

### 2.4 初始化数据库

```bash
cd myproject
python manage.py migrate
```

### 2.5 创建超级管理员

```bash
cd myproject
python manage.py createsuperuser
```

### 2.6 启动服务

开发环境：

```bash
cd myproject
python manage.py runserver 127.0.0.1:8000
```

生产环境可使用 gunicorn，具体进程管理方式按服务器规范配置。

## 3. 登录与账号操作

### 3.1 登录

1. 打开系统地址。
2. 进入 `/login/`。
3. 输入用户名和密码登录。
4. 登录成功后进入工作台。

### 3.2 注册账号

1. 打开 `/register/`。
2. 填写用户名、邮箱、角色和密码。
3. 提交后等待管理员或有权限人员审批。
4. 审批通过后，登录工作台即可看到授权菜单。

### 3.3 修改密码

1. 登录工作台。
2. 点击「修改我的密码」。
3. 按页面提示提交新密码。

## 4. 管理员配置

### 4.1 进入管理员配置

1. 使用超级管理员或有管理员配置权限的账号登录。
2. 在工作台点击「管理员配置」。

### 4.2 配置 Git 平台

填写或确认以下信息：

- GitLab 基地址，例如 `https://gitlab.example.com`。
- Git Group，例如项目所在的 GitLab Group。
- 本地工作目录，例如 `/workspace/repos`。
- Git 认证方式：优先使用 PAT，也可配置用户名和密码。

保存后，分支创建和投产追板会使用这些配置访问仓库。

### 4.3 配置 SQL 执行

填写或确认以下信息：

- SQL 仓库本地路径。
- SQL 仓库 clone 地址。
- 目标数据库主机、端口、库名、用户名和密码。
- SQL 文件阶段关键字，例如 `ddl`、`backup`、`execute`、`rollback`。
- 自动审批执行顺序，例如 `backup,ddl,execute,rollback`。

配置完成后，SQL 执行申请模块才能同步仓库、预览文件和执行脚本。

### 4.4 配置角色权限

1. 进入角色权限配置页面。
2. 选择角色。
3. 配置菜单权限，例如投产追版、投产征集、HOBO 需求登记、SQL 执行申请等。
4. 配置操作权限，例如分支任务执行、SQL 审批、投产追板执行等。
5. 配置数据范围，例如仅本人或全部。
6. 保存配置。

## 5. 用户审批

1. 使用有审批权限的账号登录。
2. 进入用户审批页面。
3. 查看待审核用户。
4. 根据申请角色和实际权限需求选择通过或拒绝。
5. 审批通过后，用户可按角色权限访问对应功能。

## 6. 工作台使用

登录后进入工作台。工作台根据当前用户权限展示可用入口：

- 投产追版：批量处理投产分支到目标分支的追板流程。
- 查询所有新建分支：汇总并执行待建分支任务。
- 投产征集：维护投产批次和投产行项。
- HOBO 需求登记：维护 HOBO 需求台账。
- SQL 执行申请：创建和审批 SQL 执行申请。
- 管理员配置：维护平台配置。

如果看不到某个入口，请检查用户审核状态和角色权限配置。

## 7. 投产征集操作

### 7.1 创建或选择投产批次

1. 进入「投产征集」。
2. 创建新的投产批次，填写投产日期、投产类型和投产分支。
3. 根据需要维护批次关联工程。

### 7.2 填写投产行项

1. 在批次下新增或编辑投产行项。
2. 填写工程、流程名、需求分支、技术负责人、业务负责人等信息。
3. 按页面要求填写参数、菜单、DIFS、流程图、部署状态、测试状态等检查项。
4. 系统会识别缺失字段并提示。

### 7.3 提交与维护

1. 确认信息完整后提交行项。
2. 如需调整，按权限进行编辑、批量更新或删除。
3. 投产行项后续会作为分支创建和投产追板的数据来源。

## 8. HOBO 需求登记操作

1. 进入「HOBO 需求登记」。
2. 新增需求登记。
3. 选择需求类型和工程。
4. 填写符合规范的需求分支名称和需求描述。
5. 保存后，可在列表中查询、编辑或删除。
6. HOBO 台账中的数据可作为新建分支任务来源。

## 9. 新建分支操作

### 9.1 预览任务

1. 进入「查询所有新建分支」。
2. 选择任务来源或筛选条件。
3. 点击预览。
4. 系统会汇总投产征集和 HOBO 台账中的待建分支。
5. 如开启远端检查，系统会检查对应分支是否已存在。

### 9.2 执行任务

1. 确认预览结果。
2. 勾选要执行的任务。
3. 点击执行。
4. 查看执行进度和结果。
5. 若失败，根据错误信息检查 Git 配置、仓库权限、分支是否已存在或命名是否符合规则。

### 9.3 定时任务

1. 在分支任务页面配置定时任务。
2. 保存定时规则。
3. 可手动触发或等待到期执行。
4. 查看定时任务运行记录。

## 10. 投产追板操作

### 10.1 使用前检查

确认以下条件：

- 当前用户具备投产追板权限。
- 管理员已配置 GitLab 地址、Group、PAT 或用户名密码。
- 本地工作目录可写，或允许系统回退到 `.runtime/repos`。
- 投产批次和行项数据已经准备好。
- 目标仓库、投产分支和目标分支存在。

### 10.2 启动追板

1. 进入 `/release-track/` 或在工作台点击「投产追版」。
2. 选择投产批次。
3. 可选填写或覆盖：
   - Tag 名称。
   - Merge 信息。
   - Tag 信息。
   - 默认目标分支。
4. 根据需要勾选：
   - Dry Run：演练模式。
   - 本次不打 Tag：只执行到指定阶段。
5. 勾选需要参与追板的项目。
6. 点击「启动追板」。

### 10.3 查看进度

启动后页面会展示：

- 当前阶段。
- 总数、已处理数、成功数、失败数、跳过数。
- 实时日志。
- 每个仓库的投产分支、目标分支、阶段、状态、原因、待合并提交和 Tag 结果。

可点击「刷新进度」查看最新状态。

### 10.4 处理 MR 审批闸口

当流程进入 MR 阶段时：

1. 在页面中查看 MR 链接。
2. 打开 GitLab MR。
3. 按组织规范完成 MR 审批和合并。
4. 回到平台页面。
5. 点击「我已完成MR，继续打Tag」。
6. 系统会检查 MR 状态并继续后续 Tag 流程。

### 10.5 常见问题

- Git clone 失败：检查 Git 地址、Group、PAT、账号密码和网络。
- 工作区不干净：清理对应本地仓库或更换工作目录。
- 分支不存在：检查投产分支、目标分支和批次数据。
- 本地试合并失败：需要先解决代码冲突。
- Tag 已存在：系统会按当前逻辑跳过或提示，按页面结果处理。

### 10.6 命令行追板

在 `myproject` 目录下可执行：

```bash
python manage.py release_track --batch-id <BATCH_ID> --config-file <配置文件路径>
```

常用参数：

- `--batch-id`：投产批次 ID，必填。
- `--config-file`：配置文件路径，用于读取 Tag、Merge 信息和目标分支映射。
- `--tag-name`：覆盖配置中的 Tag 名称。
- `--merge-message`：覆盖配置中的 Merge 信息。
- `--tag-message`：覆盖配置中的 Tag 信息。
- `--default-target-branch`：默认目标分支，默认 `master`。
- `--work-base-dir`：本地仓库工作目录。
- `--dry-run`：演练模式。

## 11. SQL 执行申请操作

### 11.1 使用前检查

确认以下条件：

- 管理员已配置 SQL 仓库路径或 clone 地址。
- 管理员已配置目标数据库连接信息。
- 当前用户具备 SQL 申请或审批权限。
- SQL 文件按投产日期或目录规范存放。

### 11.2 同步 SQL 仓库

1. 进入「SQL 执行申请」。
2. 点击仓库同步。
3. 等待同步完成。
4. 如失败，检查 SQL 仓库地址、认证和本地路径。

### 11.3 创建申请

1. 选择投产日期或 SQL 目录。
2. 浏览目录下 SQL 文件。
3. 预览需要执行的文件内容。
4. 勾选需要执行的 SQL 文件。
5. 提交申请。
6. 申请进入待审批状态。

### 11.4 审批申请

1. 使用有审批权限的账号进入 SQL 执行申请页面。
2. 查看待审批申请。
3. 检查 SQL 文件、执行阶段和风险。
4. 选择通过或拒绝。
5. 如启用自动审批能力，可按权限执行批量审批。

### 11.5 查看执行结果

审批通过后，系统会执行 SQL 并更新状态。用户可查看：

- 当前状态：待审批、审批通过、执行中、审批拒绝、执行成功、执行失败。
- 执行进度提示。
- 执行日志。
- 文件预览和已选文件。

如执行失败，应根据日志检查 SQL 语法、数据库连接、账号权限和脚本顺序。

## 12. 维护与排错

### 12.1 Git 认证失败

- 检查管理员配置中的 GitLab 基地址和 Group。
- 检查 PAT 是否有效。
- 检查用户名和密码是否正确。
- 检查账号是否有目标仓库访问权限。

### 12.2 工作目录异常

- 检查配置的 `work_base_dir` 是否存在并可写。
- 如不可写，系统会尝试使用项目下 `.runtime/repos`。
- 清理异常仓库前，应确认没有正在执行的任务。

### 12.3 数据库异常

- 检查 `MYSQL_HOST`、`MYSQL_PORT`、`MYSQL_DATABASE`、`MYSQL_USER` 和 `MYSQL_PASSWORD`。
- 检查数据库账号是否具备建表、读写和执行 SQL 的权限。
- 生产环境确认数据库密码非空。

### 12.4 权限异常

- 确认用户审核状态为已通过。
- 确认角色启用。
- 确认角色权限策略已打开对应菜单和操作权限。
- 如果数据不可见，检查对应数据范围是否为仅本人。

## 13. 文档维护

- 新增或调整页面入口。
- 修改角色权限键名。
- 修改投产追板或 SQL 执行流程。
- 修改部署方式、环境变量或依赖版本。
- 修改管理员配置项。
