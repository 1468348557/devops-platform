# DevOps Platform 需求文档

## 1. 项目背景

DevOps Platform 是面向内部研发、运维、审批人员和平台管理员的投产协作平台。系统围绕投产批次、分支创建、投产追板和 SQL 执行申请等流程，统一管理账号权限、业务数据、审批动作和执行结果，减少人工登记、人工检查和跨系统流转成本。

## 2. 项目目标

- 统一管理投产相关信息，包括投产批次、投产项目、需求分支、HOBO 需求台账等。
- 通过角色和权限策略控制不同用户可访问的菜单、可执行的操作以及可查看的数据范围。
- 支持从投产征集和 HOBO 台账中汇总待建分支需求，并提供分支任务预览、执行和定时触发能力。
- 支持投产追板流程，包括批次选择、仓库预检、MR 创建、人工审批闸口、Tag 创建和进度跟踪。
- 支持 SQL 执行申请流程，包括 SQL 仓库同步、脚本选择、申请审批、按阶段执行和日志留痕。
- 提供管理员配置入口，集中维护 GitLab、工作目录、SQL 仓库、数据库连接和执行规则等参数。

## 3. 用户角色

### 3.1 平台管理员

负责系统初始化、Git 与 SQL 平台配置、角色权限配置、用户审批、基础数据维护和异常处理。

### 3.2 运维/审批人员

负责投产批次管理、投产征集审核、分支创建执行、投产追板推进、SQL 执行审批和执行结果确认。

### 3.3 研发人员

负责注册账号、维护个人投产信息、登记 HOBO 需求、提交 SQL 执行申请，并按权限查看本人或相关数据。

## 4. 功能需求

### 4.1 账号、角色与权限

- 用户应支持注册、登录、退出和修改个人密码。
- 注册用户应绑定一个可注册的角色。
- 用户注册后默认进入待审核状态，经审批通过后才能使用对应平台功能。
- 系统应支持角色定义和角色权限策略配置。
- 权限应至少覆盖菜单权限、操作权限和数据范围。
- 超级管理员应默认拥有全部菜单、操作和数据访问能力。

### 4.2 工作台

- 用户登录后进入工作台。
- 工作台应展示当前用户名、角色和审核状态。
- 工作台根据权限展示可用功能入口，包括投产追版、查询所有新建分支、投产征集、HOBO 需求登记、SQL 执行申请和管理员配置。
- 无权限或待审核用户应看到明确提示。

### 4.3 管理员配置

- 系统应提供单例配置，用于维护 GitLab 基地址、Git Group、本地工作目录、Git 用户名密码或 PAT。
- 系统应支持配置 SQL 仓库路径、SQL 仓库 clone 地址、目标数据库连接信息、SQL 文件阶段关键字和自动审批顺序。
- 当数据库配置不可用时，系统应能使用环境变量作为 Git 运行时兜底。
- 当配置的工作目录不可写时，系统应回退到项目内 `.runtime/repos` 目录。

### 4.4 投产征集

- 系统应支持按投产日期创建投产批次。
- 每个投产批次应维护投产类型、投产分支、批次状态和创建人。
- 批次下应能关联多个工程。
- 投产行项应记录工程、需求分支、流程名、技术负责人、业务负责人、参数、菜单、流程图、部署状态、测试状态等信息。
- 系统应自动识别缺失字段，并标记行项是否填写完整。
- 投产行项应支持创建、更新、提交、删除和批量更新等操作，具体能力受权限控制。

### 4.5 HOBO 需求登记

- 系统应支持登记 HOBO 需求台账。
- 台账应记录需求类型、需求分支、工程、需求描述、申请人等信息。
- 分支名称应符合系统规定格式。
- HOBO 台账数据应作为新建分支任务来源之一。

### 4.6 新建分支与任务执行

- 系统应汇总投产征集和 HOBO 台账中的待建分支需求。
- 用户应能预览待执行分支任务。
- 系统应支持对远端仓库进行分支存在性检查。
- 用户应能按权限执行分支创建任务。
- 系统应记录任务执行状态、单项结果和错误原因。
- 系统应支持定时任务的保存、删除、触发和到期执行。

### 4.7 投产追板

- 系统应支持选择投产批次后发起投产追板。
- 用户应能覆盖 Tag 名称、Merge 信息、Tag 信息和默认目标分支。
- 系统应支持 Dry Run 演练模式。
- 系统应支持本次不打 Tag 的执行选项。
- 系统应从批次数据中构建仓库追板计划。
- 系统应拉取仓库、更新远端信息、检查目标分支和投产分支是否存在。
- 系统应统计投产分支相对目标分支的待合并提交数量。
- 系统应通过本地试合并发现冲突或失败风险。
- 系统应为可追板仓库创建 GitLab MR。
- 系统应在 MR 阶段提供人工审批闸口，等待用户完成 MR 后继续 Tag。
- 系统应在审批后检查 MR 状态，并对满足条件的仓库创建和推送 Tag。
- 系统应记录运行批次、阶段、总数、已处理数量、成功数、失败数、跳过数和各仓库执行明细。
- 页面应能实时查看日志、阶段状态、仓库状态和 MR 链接。

### 4.8 SQL 执行申请

- 系统应支持同步 SQL 仓库。
- 用户应能按投产日期或目录浏览 SQL 文件。
- 用户应能预览 SQL 文件内容。
- 用户应能勾选 SQL 文件并创建执行申请。
- 申请应记录申请日期、SQL 目录、选择文件、申请人、状态、审批人和执行日志。
- 审批人员应能通过或拒绝申请。
- 系统应支持自动批量审批能力，具体能力受权限和配置约束。
- 审批通过后，系统应按配置的阶段关键字和顺序执行 SQL。
- 系统应提供执行进度、执行结果和日志查看能力。

## 5. 非功能需求

### 5.1 权限与安全

- 生产环境必须配置非默认 `DJANGO_SECRET_KEY`。
- 生产环境必须配置 MySQL 密码。
- Git 与数据库凭据应通过配置或环境变量维护，不应写入代码。
- 日志和错误信息中应尽量脱敏 Git Token、密码等敏感内容。
- 生产环境应启用 HTTPS、安全 Cookie 和合适的 Host 限制。

### 5.2 可维护性

- 系统应使用 Django 标准应用结构组织代码。
- 关键业务操作应通过数据库记录运行状态和结果。
- 业务配置应集中在管理员配置或环境变量中，减少硬编码。
- 文档应随路由、权限键名、部署命令和业务流程变更同步更新。

### 5.3 可追踪性

- 投产追板和 SQL 执行应记录执行人、开始/结束时间、状态、错误信息和明细。
- 分支任务执行应记录每项结果，便于问题追踪和重试。
- 用户审批、角色权限、管理员配置应保留更新人和更新时间等基础信息。

## 6. 技术与环境需求

- 后端框架：Django 5.2.x。
- 数据库：MySQL，使用 PyMySQL 以 MySQLdb 兼容方式接入。
- 部署运行：支持 Django 管理命令和 gunicorn。
- 配置方式：支持 `myproject/.env` 与操作系统环境变量。
- 主要应用：`accounts`、`branch_create`、`release_track`、`sql_execute`。

## 7. 主要页面与路由

| 路径前缀 | 模块 | 说明 |
| --- | --- | --- |
| `/` | 账号与工作台 | 登录后工作台、注册、审批、管理员配置、角色权限 |
| `/branch-create/` | 分支与投产登记 | 新建分支、投产征集、HOBO 台账、分支任务和定时任务 |
| `/release-track/` | 投产追板 | 批次追板、MR、Tag、进度查询和审批继续 |
| `/sql-execute/` | SQL 执行 | SQL 仓库同步、申请、审批、执行和日志 |
| `/admin/` | Django Admin | 后台管理入口 |

## 8. 验收标准

- 用户可完成注册、审批、登录和按角色访问工作台。
- 管理员可维护 Git、SQL、数据库和权限相关配置。
- 投产批次、投产行项和 HOBO 需求可正常创建、修改、查询。
- 分支任务可从投产征集和 HOBO 数据中汇总、预览并执行。
- 投产追板可完成批次选择、Dry Run、MR 创建、审批继续、Tag 和结果展示。
- SQL 执行申请可完成仓库同步、文件选择、申请审批、执行和日志查看。
- 所有关键流程均有状态和错误信息留痕。
